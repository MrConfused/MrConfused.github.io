二、项目总结

1.版本控制

  • 版本控制系统 Version Control System (VCS)
    • 源码管理Source Code Manager (SCM)
      • CocoaPods：Package Manger (PM)，对每个独立的包进行管理管理，被 PM 接管的依赖库的文件，通常会在 Git 的 .ignore 文件中选择忽略它们。
      • Git和SVN：对单个文件进行管理

1.2 Package Manger

PM 工具基本围绕这个两个文件来现实包管理：

  • 描述文件：声明了项目中存在哪些依赖，版本限制；
  • 锁存文件（Lock 文件）：记录了依赖包最后一次更新时的全版本列表。
Key File Git submodule CocoaPods
描述文件 .gitmodules Podfile
锁存文件 .git/modules Podfile.lock

1.3 Git Submodule

Git Submodules 将单独的 git 仓库以子目录的形式嵌入在工作目录中。

.gitmodules

Git submodule 是依赖 .gitmodules 文件来记录子模块的 path 和 url 以及模块名称的基本信息。 而且 .gitmodules 文件中的 path 也会被 git 直接 ignore 掉。

.git/modules目录

每个 Submodule Repo 的 commit 信息是记录在 .git/modules 目录下。

1.4 CocoaPods

CocoaPods 是iOS开发的第三方库的依赖管理工具。可以定义自己的依赖关系。

Podfile

Podfile 是一个文件，以 DSL（其实是 Ruby 语法）来描述依赖关系，用于定义项目所需要使用的第三方库。该文件支持高度定制。

Podfile.lock

它记录了需要被安装的 Pod 的每个已安装的版本。将 Podfile.lock 文件加入到版本控制中，这有助于整个团队的一致性。

Manifest.lock

这是每次运行 pod install 命令时创建的 Podfile.lock 文件的副本。

项目中经常遇到的错误：The sandbox is not in sync with the Podfile.lock，这是因为 Manifest.lock 文件和 Podfile.lock 文件不一致所引起。由于 Pods 所在的目录并不总在版本控制之下，这样可以保证我们在运行 App 之前都能更新各自项目中 Pods，否则 App 可能会 crash，或者在一些不太明显的地方编译失败。

Master Specs Repo

CocoaPods 通过官方的 Spec 仓库来管理这些注册的依赖库。但是随着不断新增的依赖库导致 Spec 的更新和维护成为了使用者的包袱。好在这个问题在 1.7.2 版本中已经解决了，CocoaPods 提供了 Mater Repo CDN ，可以直接 CDN 到对应的 Pod 地址而无需再通过本地的 Spec 仓库了。

同时在 1.8 版本中(我们的项目是1.8.4)，官方默认的 Spec 仓库已替换为 CDN，其地址为 https://cdn.cocoapods.org。

2.Ruby

CocoaPods 是通过 Ruby 语言实现的（它本身就是一个 Gem 包）。

RVM 和 rbenv 都能提供不同版本的 Ruby 环境管理和切换。

2.1 RubyGems

RubyGems 是 Ruby 的一个包管理工具，这里面管理着用 Ruby 编写的工具或依赖（Gem）。

当我们使用 gem install xxx 时，会通过 rubygems.org 来查询对应的 Gem Package。

而 iOS 开发中的很多工具都是 Gem 提供的，例：Bundler，fastlane，CocoaPods 等。

在默认情况下 Gems 总是下载 library 的最新版本，这无法确保所安装的 library 版本符合我们预期。因此我们还缺一个工具。

2.2 Bundler

Bundler 是管理 Gem 依赖的工具，可以隔离不同项目中 Gem 的版本和依赖环境的差异（其本身也是一个 Gem）。

Bundler 通过读取项目中的依赖描述文件 Gemfile ，来确定各个 Gems 的版本号或者范围，来提供了稳定的应用环境。

当我们使用 bundle install 它会生成 Gemfile.lock 将当前 librarys 使用的具体版本号写入其中。之后，他人再通过 bundle install 来安装 libaray 时则会读取 Gemfile.lock 中的 librarys、版本信息等。

Gemfile

Bundler 依据项目中的 Gemfile 文件来管理 Gem，（而 CocoaPods 通过 Podfile 来管理 Pod）。

Gemfile 的 DSL 写法和 Podfile 如出一辙。

那什么情况会用到 Gemfile 呢 ？

CocoaPods 每年都会有一些重大版本的升级，前面聊到过 CocoaPods 在 install 过程中会对项目的 .xcodeproj 文件进行修改，不同版本其有所不同，这些在变更都可能导致大量 conflicts，处理不好，项目就不能正常运行了。

如果项目是基于 fastlane 来进行持续集成的相关工作以及 App 的打包工作等，也需要其版本管理等功能。

3.2 使用 rbenv 管理 Ruby 版本

安装成功后，我们让其生效：

  • rbenv global 2.6.0 # 默认使用 2.6.0

  • rbenv shell 2.6.0 # 当前的 shell 使用 2.6.0, 会设置一个 RBENV_VERSION 环境变量

  • rbenv local 2.6.0 # 当前目录使用2.6.0, 会生成一个 .rbenv-version 文件

输入上述命令后，可能会有报错。rbenv 提示我们在 .zshrc 中增加一行 eval "$(rbenv init -)" 语句来对 rbenv 环境进行初始化。如果报错，我们增加并重启终端即可。
